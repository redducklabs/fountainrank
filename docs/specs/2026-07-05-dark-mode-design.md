# Dark Mode (web + mobile + shared dark basemap) — Design

**Date:** 2026-07-05
**Issue:** #18 — Dark mode (app-wide, incl. dark basemap flavor)
**Status:** Draft (pending Codex review + owner review)
**Scope decision:** All-in-one — web UI, mobile UI, and the shared dark basemap in one spec.

---

## 1. Summary

Add an app-wide dark theme to FountainRank across **web** (Next.js 16 + Tailwind v4),
**mobile** (Expo/React Native), and the **shared self-hosted Protomaps basemap**. Today there is
**no theming layer anywhere**: web uses ~148 hardcoded hex literals as Tailwind arbitrary utilities
(`bg-[#0A357E]`) across 59 files; mobile has a single flat light `theme.ts` imported directly into
37 files; the basemap style is generated at CI time hardcoded to `namedFlavor("light")` and baked
into the web image at build time. This design introduces a real semantic-token theming layer on both
clients, a persisted **System / Light / Dark** switch, a runtime-swappable dark basemap, and a
second dark-tuned pin asset set.

### Goals

- Three theme states everywhere: **System** (default, follows OS), **Light**, **Dark**; explicit
  choice persists.
- A semantic **token layer** on web (Tailwind v4 `@theme inline` + `.dark` override) and mobile
  (light/dark palettes behind a provider) so colors are no longer scattered hex literals.
- A **dark basemap** (`style.dark.json` + dark sprite) selected at runtime, shared by web + mobile.
- Dark-tuned **map pins** (a second baked raster set) so contrast holds on the dark basemap.
- **WCAG AA** contrast in both themes; status stays encoded by shape/glyph, never color alone.

### Non-goals

- No backend change: no schema change, no API change, **no api-client regeneration**.
- No re-tiling of `planet.pmtiles` (tile data is flavor-agnostic; only the style JSON differs).
- No redesign of the brand identity — the navy/gold/cyan brand palette is retained; dark mode
  re-tones surfaces and text, not the brand.
- No per-component "themes" beyond light/dark (no user-custom accent colors, etc.).

## 2. Locked decisions

| Decision | Choice |
|---|---|
| Scope | All-in-one: web + mobile + shared basemap |
| Web theming model | Full semantic token system (Tailwind v4 `@theme inline` + `.dark` class) |
| Toggle model | 3-state: **System** (default) / Light / Dark; persisted |
| Web provider | **`next-themes@0.4.6`** (React 19 compatible; battle-tested no-flash SSR) |
| Map basemap swap | `map.setStyle(darkUrl)` + **re-install overlay** (preserve camera) |
| Toggle placement | Web: `SiteHeader` + account page. Mobile: account/profile tab |
| Pins on dark | Second dark-tuned baked set generated by `scripts/gen-pin-assets.py` |

## 3. Token system & palette

### 3.1 Semantic tokens

A single canonical set of **semantic** tokens (not raw brand names) is defined once per client. Each
token has a light value and a dark value. Brand colors from `docs/style-guide.md` seed the light
values; dark values are re-toned surfaces/text plus brightened map colors.

**Brand constants (unchanged, both themes):** Navy `#0A357E`, Blue-mid `#0C44A0`, Royal `#0E4DA4`,
Crown gold `#F2C200`, Water cyan `#5FC5F0`.

**Proposed palette** (dark values are a starting point; the a11y task tunes every pair to WCAG AA
against real screenshots — see §9):

| Semantic token | Light | Dark (proposed) | Notes |
|---|---|---|---|
| `background` | `#FFFFFF` | `#0B1220` | Page/content base |
| `surface` | `#F8FAFC` | `#111A2E` | Cards, panels, list rows |
| `surface-raised` | `#FFFFFF` | `#16213A` | Elevated (detail panel, menus) |
| `foreground` | `#0F172A` | `#E6EDF7` | Primary text |
| `muted` | `#475569` | `#9FB0C7` | Secondary text |
| `border` | `#E2E8F0` | `#26324A` | Hairlines, dividers |
| `brand` | `#0A357E` | `#0A357E` | Navy; brand bands keep it |
| `brand-mid` | `#0C44A0` | `#2A5CC0` | Gradient middle |
| `brand-royal` | `#0E4DA4` | `#2A5CC0` | Gradient bottom / primary blue |
| `accent-gold` | `#F2C200` | `#F2C200` | Gold accent (both themes) |
| `water` | `#5FC5F0` | `#5FC5F0` | Decorative accents |
| `danger` | `#B91C1C` | `#F87171` | Errors (brightened on dark) |
| `on-brand` | `#FFFFFF` | `#FFFFFF` | Text on brand bands |

**Map tokens** — these are **JS constants** consumed by MapLibre paint (web: a new
`web/lib/map/colors.ts` keyed by theme; mobile: the `theme.ts` palette), **not** CSS `@theme` vars,
since map paint is set in JavaScript and mobile has no CSS layer. Brightened so pins/labels hold
contrast on the dark basemap:

| Map token | Light | Dark (proposed) |
|---|---|---|
| `map-cluster` | `#0C44A0` | `#4C82F0` |
| `map-cluster-stroke` | `#FFFFFF` | `#0B1220` |
| `map-cluster-count` | `#FFFFFF` | `#FFFFFF` |
| `map-pill-bg-asset` | `pill-bg.png` | `pill-bg-dark.png` |
| `map-pill-text` | `#0A357E` | `#E7F0FF` |
| `map-halo` | `#0C44A0` | `#5FC5F0` |

### 3.2 Brand bands in dark mode

Several web surfaces are the navy **brand gradient** shell (hero band on `/`, footer, account
gradient). These already have adequate white-on-navy AA contrast and read fine on a dark page, so in
dark mode they are **retained** (optionally deepened one step). Dark mode primarily re-tones the
**white content surfaces** and slate text — not the brand bands.

## 4. Web architecture

### 4.1 Token layer (`web/app/globals.css`)

```css
@import "tailwindcss";

/* Class-based dark variant (overrides v4's default prefers-color-scheme) */
@custom-variant dark (&:where(.dark, .dark *));

:root {
  --background: #ffffff;
  --surface: #f8fafc;
  --foreground: #0f172a;
  /* ...all semantic + map tokens (light values)... */
}

.dark {
  --background: #0b1220;
  --surface: #111a2e;
  --foreground: #e6edf7;
  /* ...dark overrides... */
}

/* Expose tokens as Tailwind utilities whose values follow the CSS vars */
@theme inline {
  --color-background: var(--background);
  --color-surface: var(--surface);
  --color-foreground: var(--foreground);
  /* ...one per token... */
}
```

This generates real utilities (`bg-background`, `text-foreground`, `bg-surface`, `text-map-pill`,
…) whose resolved values flip when `.dark` is present on `<html>`. Confirmed against current Tailwind
v4 docs (`@custom-variant dark`, `@theme inline`).

### 4.2 Provider + no-flash (`next-themes`)

- Add `next-themes@0.4.6` to `web/package.json`.
- A client `<ThemeProvider attribute="class" defaultTheme="system" enableSystem>` wraps the app.
- `web/app/layout.tsx`: add `suppressHydrationWarning` to `<html>`; `next-themes` injects its own
  pre-hydration script that sets the `.dark` class **before first paint** (no FOUC) and resolves
  `system` from `prefers-color-scheme`.
- Persistence: `next-themes` writes the choice to `localStorage` (key `theme`) by default.

### 4.3 Toggle component

A new `ThemeToggle` (3-state: System / Light / Dark) using `useTheme()` from `next-themes`. Rendered
in `SiteHeader` (always reachable) and mirrored on the account page. It must be **hydration-safe**
(render a stable placeholder until mounted, per `next-themes` guidance) and keyboard-accessible with
visible focus + `aria` state. Document it in `docs/style-guide.md`.

### 4.4 Migration of hardcoded hex

Replace the ~148 arbitrary hex utilities across 59 files with token utilities. The palette is a small
closed set, so this is mechanical but broad. Priority surfaces:

- **Map:** `web/lib/map/layers.ts` (see §5.2), `web/components/map/MapStates.tsx`,
  `MapBrowserLoader.tsx`, `AddFountainPanel.tsx`, `web/components/map/placement-map.ts`.
- **Detail:** `FountainDetail.tsx`, `DetailOverlay.tsx`, `FountainDetailTabs.tsx`, `AttributeChips.tsx`,
  `Stars.tsx`, `StarGroup.tsx`.
- **Leaderboard:** `LeaderboardRows.tsx`, `LeaderboardControls.tsx`.
- **Brand / chrome:** `SiteHeader.tsx`, `app/page.tsx`, `terms/page.tsx`, `privacy/page.tsx`,
  `account/page.tsx`, `account/fountains/page.tsx`.

Existing component tests that assert on `bg-[#...]` class strings are updated to the token utilities.

## 5. Web map (runtime basemap swap)

### 5.1 Style URL derivation (`web/lib/map/style.ts`)

`BASEMAP.styleUrl` (from `NEXT_PUBLIC_BASEMAP_STYLE_URL`) points at `…/style.light.json?v=N`. Add:

```ts
export function styleUrlFor(theme: "light" | "dark"): string {
  // derive dark URL from the configured light URL by filename convention,
  // preserving the ?v= cache-bust query
  return BASEMAP.styleUrl.replace("style.light.json", `style.${theme}.json`);
}
```

No second env var; the dark URL is the light URL with the flavor filename swapped. If the configured
URL does not contain `style.light.json`, `styleUrlFor("dark")` returns the original (safe fallback →
light basemap), so a misconfiguration degrades gracefully rather than 404-ing the map.

### 5.2 `installOverlay` extraction + `setStyle`

Refactor the current `map.on("load")` body in `web/components/map/MapBrowser.tsx` into a reusable
`installOverlay(map, resolvedTheme)` that:

1. loads the theme-appropriate pin images + pill-bg and `addImage`s them,
2. adds the `fountains` source (re-applying the current `pins` FeatureCollection),
3. adds all layers (cluster/count/pins/pill/halo/selected) built from theme-aware factories,
4. re-wires click/hover handlers and the active-id filter.

On initial map creation it runs on `load` (as today). On **theme change**, a new effect calls
`map.setStyle(styleUrlFor(resolvedTheme))` and re-runs `installOverlay` on the next `style.load`
(MapLibre `setStyle` wipes custom sources/layers/images). Camera (center/zoom/bearing/pitch) is
preserved across `setStyle`. The map is **not** rebuilt, so no flash and no geolocation re-trigger.

- The resolved theme comes from `next-themes` `useTheme().resolvedTheme` (so `system` resolves to
  light/dark). The map effect depends on `resolvedTheme`.
- Guard against `setStyle` races: track a style-generation counter (mirroring the existing
  `loadSeqRef` pattern) so a rapid light→dark→light toggle can't re-install stale layers.

### 5.3 Theme-aware layer factories (`web/lib/map/layers.ts`)

`clusterCircleLayer`, `clusterCountLayer`, `pillLayer`, `selectedHaloLayer` receive a resolved
`MapColors` object from `web/lib/map/colors.ts` (keyed by resolved theme) — a plain TS constant, so
factories stay pure and unit-testable with no `getComputedStyle`/DOM dependency. Pin/pill
`icon-image` names become theme-suffixed (e.g. `pin-standard` vs `pin-standard-dark`, `pill-bg` vs
`pill-bg-dark`).

## 6. Mobile architecture

### 6.1 Palette split + provider

- Split `mobile/theme.ts` into `lightColors` / `darkColors` (same keys), keep `spacing`/`typography`
  shared.
- Add a `ThemeProvider` (React context) that merges `useColorScheme()` (react-native OS setting)
  with a persisted override (AsyncStorage key `theme`), exposing `useTheme(): { colors, spacing,
  typography, scheme, preference, setPreference }`.
- Rewire the 37 direct importers of `colors`/`typography` to `useTheme()`. Since `StyleSheet.create`
  is static, components read colors from the hook and either build styles inline or via a
  `makeStyles(theme)` helper.

### 6.2 System chrome

- `StatusBar` (`mobile/app/_layout.tsx`) becomes theme-aware (`style` from resolved scheme; the two
  hardcoded `style="dark"` usages).
- React Navigation `ThemeProvider` gets `DarkTheme`/`DefaultTheme` (or custom equivalents) so
  navigators, headers, and the tab bar honor the theme; `TAB_INACTIVE_COLOR` moves into the palette.
- Splash/adaptive-icon backgrounds in `app.config.ts` (`#ffffff`) are left as-is for launch (native
  splash can't be themed at runtime) unless a trivial dark launch background is warranted — flagged
  as a low-priority tuning item, not a blocker.

### 6.3 Toggle

A theme control (System / Light / Dark) on the account/profile tab, using `useTheme().setPreference`.

### 6.4 Mobile map (`mobile/components/map/FountainMap.tsx`, `app.config.ts`)

- `app.config.ts` `basemapStyleUrl` stays the light URL; mobile derives the dark URL by the same
  filename convention (a shared helper mirroring web's `styleUrlFor`).
- `FountainMap` selects `mapStyle` by resolved scheme and re-renders on change; MapLibre-RN handles
  the style swap. Layer colors (cluster/pill/halo/search-marker) read from `useTheme().colors` map
  tokens. Pin `require()`s (`mobile/assets/pins/*.png`) become theme-keyed (`*-dark.png`).

## 7. Shared basemap pipeline (`.github/workflows/basemap-upload.yml`)

- Add a **dark flavor pass** to the existing style-generation step: run the Node generator a second
  time with `namedFlavor("dark")`, emitting `style.dark.json`, and set its `sprite` to
  `https://${CDN_HOST}/sprites/v4/dark`.
- Generate + upload a **dark sprite** (`sprites/v4/dark`) alongside the existing light sprite.
- Upload `style.dark.json` to the Spaces bucket next to `style.light.json` (both public-read → CDN).
- The vector source (`/tiles/planet.json`) is unchanged and shared by both styles.
- The existing `BASEMAP_STYLE_VER` cache-bust applies to both style files (clients request
  `style.<flavor>.json?v=N`).
- `basemap-janitor.yml` cleanup is extended to keep both flavors.

**Rollout ordering:** the basemap workflow must run and publish `style.dark.json` **before** a web
image that can request it reaches users. Because the dark URL is derived at runtime (not baked), a
web deploy that ships the toggle before `style.dark.json` exists would 404 the dark basemap; the
`styleUrlFor` fallback (§5.1) covers a misconfigured URL but not a missing dark file. The plan
sequences the basemap upload first.

---

## 8. Pin asset generation (`scripts/gen-pin-assets.py`, `scripts/assets/gen_unrated_pin.py`)

- Extend the Pillow generators to emit **dark-tuned variants**: brighter/lighter fills + a stronger
  contrast outline so pins pop on dark land. Output `pin-standard-dark.png`, `pin-selected-dark.png`
  (web only), `pin-gold-dark.png`, `pin-broken-dark.png`, `pin-unrated-dark.png`, and
  `pill-bg-dark.png`.
- Commit regenerated assets to **both** `web/public/pins/` and `mobile/assets/pins/` (mobile has no
  `pin-selected`). Because they are script-generated, the light and dark sets stay in sync.
- The broken-slash and crown remain baked per-variant (still shape-encoded for status), just
  re-toned for dark contrast.

## 9. Accessibility & contrast

- Every text/background token pair meets **WCAG AA** (4.5:1 body, 3:1 large/UI) in both themes.
- Map: pin fills, cluster bubbles, pill text, and the broken-slash must clear AA-equivalent contrast
  on the dark basemap land color; the a11y task verifies against real screenshots and tunes the
  proposed dark values in §3.1.
- Status is encoded by **shape/glyph** (broken-slash, crown), never color alone — already true and
  preserved.
- Toggle control is keyboard-operable with visible focus and correct `aria` semantics.

## 10. Testing & verification

**Host constraint:** JS local checks are blocked on the Windows dev host
(`ERR_PNPM_ABORTED_REMOVE_MODULES_DIR_NO_TTY`); `CI=true` must never be set (destructive purge). Web
+ mobile JS is verified via **CI's `workspace-js` job** and browser-visual in CI, not locally.

- **Web unit (vitest):** provider resolution (system/override/persist), toggle behavior + hydration
  safety, presence of token utilities, `styleUrlFor` derivation + fallback, theme-aware layer
  factories, and updated component tests asserting token classes instead of hex.
- **Mobile:** `ThemeProvider` merge logic (system + persisted override), `useTheme` consumers,
  type-check + lint via CI (mobile eslint/React-Compiler rules are CI-only).
- **Basemap:** the workflow change produces a syntactically valid `style.dark.json` (JSON parse +
  spec sanity) and a dark sprite.
- **Backend:** untouched — existing suite unaffected; no api-client regen.
- **Manual/CI visual:** light and dark screenshots of `/`, detail panel, leaderboard, and the map
  (both basemaps + pins) to confirm contrast and the no-flash behavior.

## 11. Work breakdown (for the implementation plan)

Ordered so each step is independently reviewable and the basemap exists before clients request it:

1. **Basemap dark flavor** — `basemap-upload.yml` generates + uploads `style.dark.json` + dark
   sprite; janitor updated. (Ships first.)
2. **Dark pin assets** — extend generators; commit dark PNGs to web + mobile.
3. **Web token layer** — `globals.css` `@custom-variant` + `:root`/`.dark` + `@theme inline`;
   style-guide token table.
4. **Web provider + toggle** — add `next-themes`, provider in layout, `ThemeToggle` in header +
   account, no-flash.
5. **Web hex → token migration** — the 59-file sweep + test updates.
6. **Web map theming** — `styleUrlFor`, `installOverlay` extraction, `setStyle` swap, theme-aware
   layer factories + dark pins.
7. **Mobile palette split + provider** — `theme.ts` light/dark, `ThemeProvider`, `useTheme`, rewire
   37 importers, StatusBar + React Navigation.
8. **Mobile map theming** — dark basemap derivation, layer tokens, dark pins.
9. **A11y/contrast pass** — verify + tune all token pairs and map colors to AA against screenshots.
10. **Docs** — `docs/style-guide.md` (dark tokens, ThemeToggle), update the dark-mode-ready notes.

## 12. Risks & mitigations

- **`setStyle` re-install races** — mitigated by a style-generation counter and re-installing only
  on the latest `style.load`.
- **Missing `style.dark.json` at runtime** — sequenced first (§7); `styleUrlFor` degrades to light
  on a URL that lacks the `style.light.json` marker.
- **Large migration diff** — mechanical but touches most components; mitigated by the closed palette
  and staged tasks (token layer landing before the sweep).
- **Mobile splash can't theme at runtime** — accepted; launch splash stays light, in-app chrome
  themes.
- **New dependency (`next-themes`)** — pinned `0.4.6`, React 19 compatible, well past the CI
  minimum-release-age gate; subject to pnpm-audit like any dep.

## 13. Out of scope / follow-ups

- Native dark splash screens (if desired later).
- User-custom accent colors or additional themes beyond light/dark.
- Any backend/data changes (none required).
